Git基本操作

基本操作

检查当前文件状态

使用git status命令可以列出工作区、暂存区和本地仓库之间状态不同的文件，如果使用了-ignored选项，还可以列出工作区中已忽略的文件，主要包含以几种：

  • Changes to be committed：指暂存区与本地仓库之间存在差异的文件列表
    • commit命令：暂存区=>本地仓库
    • restore –staged命令：本地仓库=>暂存区
  • Changes not staged for commit：指工作区与暂存区之间存在差异的文件列表
    • add 命令：工作区=>暂存区
    • restore命令：暂存区=>工作区
  • Untracked files：未追踪的文件，通过add命令添加到暂存区，如果文件在.gitignore中设置为忽略，则add命令不会添加
  • Ignored files：使用–ignored选项后列出以忽略的文件，可以通过-f选项强制添加到暂存区

使用-s选项可以列出文件状态简写模式，包括??，A，M，D，等，??表示Untracked files，A表示新增文件，M表示文件修改，D表示文件被删除。输出的第一列是暂存区与本地仓库之间相比较，第二列是工作区与暂存区之间相比较。

添加：工作区=>暂存区

git add .命令可以更新工作区中的所有文件，包括已删除的文件和新增的文件，也可以通过git add file 或git add *.py添加和匹配特定的文件。

  • -n，–dry-run选项：用于测试，输出该命令会add的文件列表，但是实际上不会添加文件

  • -u，–update选项：更新文件，不包括Untracked files

  • -A，–all选项：添加所有文件，与git add .相似

  • -f选项：强制添加已忽略的文件

删除和移动文件

普通的删除命令和移动命令只能改变工作区的内容，git rm file和git mv file同时在工作区和暂存区中进行删除和移动，相当于执行删除和移动指令后再自动执行add命令

git rm –cached可以仅删除暂存区文件而保留工作区。

提交：暂存区=>本地仓库

git commit命令将暂存区内容提交到本地仓库，每一次提交都会产生一个记录，不像工作区和暂存区都是临时的内容。

  • -m选项：为commit添加注释
  • -a选择：相当于提交前自动执行git add -u，不能添加新增文件
  • -amend选项：自动打开编辑器，在交互界面修改，并重新提交

使用amend选项不仅修改了提交信息，还自动修改了提交时间，第一行为创建commit的时间，第二行为最后一次提交的时间。

撤销：本地仓库=>暂存区=>工作区

git restore命令可以将暂存区的内容替换工作区的内容，使用–staged选项可以将当前本地仓库的内容（HEAD）替换暂存区的内容。

git reset commit-hash命令可以将HEAD指针移动到指定的commit上，通常用于撤销commit，同时还可以替换暂存区和工作区，具有以下模式

  • –hard：同时重置暂存区和工作区
  • –soft：既不重置暂存区也不重置工作区
  • –mixed(默认)：只重置暂存区，保留工作区
  • –merge：暂未使用
  • –keep：重置暂存区和工作区，但是如果文件在本地修改了，则终止reset

  • git branch -D dev：强制删除未合并的分支dev
  • git branch -c dev：复制分支（复制reflog）

Git内部原理

初始化仓库

使用git init可以将当前目录初始化成一个git仓库，然后就可以通过git命令进行版本控制。

初始后会生成一个隐藏目录.git，所有的数据都会保存到这个目录中，如下：

  • description 文件仅供 GitWeb 程序使用，所以不用关心这些内容。
  • config 文件包含了项目特有的配置选项.
  • info 目录保存了一份不希望在.gitignore 文件中管理的忽略模式 (ignored patterns) 的全局可执行文件。
  • hooks 目录保存了客户端或服务端钩子脚本，比如常见的pre-commit。

剩下的文件又可以分为git对象和对象的引用（指针）两大类，是git中的核心概念。

git对象

git对象都储存在目录objects中，以hash值作为文件名称，取前两位作为目录名称进行组织，可以分为以下三类：

  • blob：文件对象，将文件添加到暂存区时产生。
  • tree：目录对象，文件的集合。
  • commit：commit对象，每次commit时产生，每个commit对象引用一个tree对象，表示该版本的文件集合，除此之外，还包括提交者、提交日期等内容。

file对象都是通过文件的hash值进行命名的，这样只需要比较hash值就能判断文件是否相同，相同的文件不会重复储存，如果多个版本中都包含同一个文件，那么最终引用的都是同一个对象。计算文件的hash值时，仅包括文件的内容，不包括文件的元信息。

tree对象的内容是引用的文件对象和目录对象，一般不会涉及到，其中储存了文件的hash值和部分元信息，包括实际的文件名，权限等，但是不包括文件的最近修改时间。

commit对象也有一个hash值作为名称，部分操作需要指定特定commit对象时，就需要用到这个hash值，不用完整地输入hash值，前几位即可，只要保证唯一性就行。由于commit对象包括了时间信息，几乎不可能存在两个相同的commit对象。

查看git对象内容

git对象不能直接用编辑器查看，使用git cat-file命令可以查看git对象的内容，使用之前必须知道git对象的hash值。

  • -p选项：显示内容
  • -s选项：显示大小
  • -t选项：显示类型

引用

refs目录中，有heads和tags两个目录，分别储存分支和标签的指针，文件名就是对应的分支名称和标签名称，文件内容是对应的commit对象名称，引用本质上就是commit对象的一个便于记忆的别名。

在.git目录中还有一个HEAD文件，HEAD是对分支的引用。

HEAD指向分支，分支指向commit对象，切换分支时，只需要改变HEAD的值即可，提交或撤销提交时，只需要改变当前分支的值即可，HEAD的最终指向自然会跟着改变。

对某一分支来说，commit是线性提交的，commit对象也存储了上一个commit对象的哈希值，如果我们需要上一次或上几次commit对象的哈希值，可以通过查询日志的方式获取上一次commit的哈希值，也可以通过HEAD^、HEAD^^或HEAD1、HEAD2直接引用。

引用日志

引用日志（reflog）比日志（log）的范围更广，比如说通过commit提交了一个记录，然后发现存在问题又用reset命令撤销了，通过log命令是查不到相关记录的，但是reflog命令仍然可以查到相关的记录，包括被撤销的commit对象的哈希值。

即使没有引用日志，也照常可以使用git log命令，因为日志实际上是储存在commit对象中的，跟引用日志没有直接关系，引用日志只会影响垃圾回收的行为。

涉及到commit对象的相关命令都会被记录下来，被删除的commit不会立即回收，如果误删除了某个commit，在log中找不到了，还可以尝试在reflog中找一找。

可以看到，在不同分支上进行的commit会记录到对应的文件中，同时还会记录到log/HEAD（主reflog）中，因此分支中的commit被引用了两次，即使删除了分支仍然还是存在引用，会影响到垃圾回收。

垃圾回收

git对象对内容进行hash计算，最小单元是文件。即使两个文件只有很小的差异，仍然会创建两个完整的对象，消耗双倍的空间，而在版本控制中，这是很常见的情况。

git不会立即删除不再需要的git对象，git对象数量只增不减，只有垃圾回收时才会删除冗余的对象。例如，当把文件加入暂存区时创建了一个git对象，更新文件时又会重新创建一个git对象，一个文件在提交前暂存了好几次是常见的。

当撤销一个commit时，我们不希望立即从数据库中删除，希望能在一定时间范围内提供恢复的途径，所以git中有reflog的概念，涉及到commit的操作会记录到reflog中，垃圾回收时不会删除引用日志中存在的对象。

基于上述原因，我们需要git gc来管理git对象。git gc不仅仅是删除冗余对象，包含了以下工作：

  • pack-refs：将refs/heads和ref/tags目录下的引用打包到一个文件中（packed-refs）
  • reflog expire：默认情况下清除超过90天的reflog记录，以及清除超过30天且不再需要的reflog记录（比如说提交一个commit，又将其删除了，那么30天后，与这个commit相关的reflog记录会被删除）
  • repack：将日志和引用日志中的对象打包，其余的以松散对象的形式存在，不会被立即删除，由于git对象中存在大量重复内容，打包后会节省大量的空间。
  • prune：默认情况下删除超过14天的松散对象，主要针对暂存时产生的git对象，既没有被日志引用，也没有被引用日志引用，但是作为松散对象仍然可以保留14天。

也就是说，一个commit从日志中删除，30天后，从引用日志中删除，不再被打包，以松散对象的形式存在，显然已经超过14天，立即被彻底删除。而对于日志中存在的commit，只是90天后从引用日志中删除记录，永远不会被删除。

Nginx配置

配置文件

默认情况下，配置文件名为nginx.conf，并放在目录：/usr/local/nginx/conf, /etc/nginx, 或 /usr/local/etc/nginx 中。笔者在termux中安装了nginx，配置文件在目录/etc/nginx下。

配置文件目录的入口是nginx.conf，由指定的指令控制的模块组成。一个简单的指令由空格分隔的名称和参数组成，并以分号(;)结尾。块指令具有与简单指令相同的结构，但不是以分号结尾，而是以大括号({和})包围的一组附加指令结束。 如果块指令可以在大括号内部有其他指令，则称为上下文(例如：events，http，server和location)。

如果配置文件的内容太多，或者是为了配置文件的结构更加清晰，可以在配置目录中建立一个名为conf.d的目录，每一个server块都写到一个单独的文件中，并放到conf.d目录下，最后再nginx.conf中通过include引入。

http块

http块是对http服务器相关的配置，http块配置的指令包括：文件引入、MIME-TYPE 定义、日志自定义、连接超时时间、单链接请求数上限等。

在http块中的server块相当于一台虚拟主机，有自己的服务器名字和监听的端口号，每个server块中还有多个location块，可以根据url的路径进行匹配。error_page表示服务器出错时返回的页面，错误代码为404时返回页面404.html，表示资源不存在，错误代码为502、502、503、504时返回页面50x.html，表示服务器错误，这些都是nginx的默认处理方式。

请求分发

如果要在一台主机上部署多个server，可以通过域名来进行区分，也可以通过端口号来进行区分，将请求发送到对应的server中。如果将不同的主机部署到不同的端口上，就不得不开放多个端口号，而且一般来说http端口号采用80，https端口号采用443，如果服务器在其他端口号上，访问不是很方便。

以上前两个都是静态页面，分别挂载在/home和/home/jsom上，alias表示资源实际的位置，autoindex表示允许列举出目录，默认情况下当请求路径是一个目录时，那么在该目录下寻找index.html作为返回页面，如果没找到则返回404。开启autoindex后，请求目录时则返回下目录下的所有文件详情。

autoindex_format表示请求一个目录时返回的格式，默认为html格式，页面下的每个文件都是超链接，便于在浏览器中浏览。也可以设置为json格式，json格式比html格式更容易在程序中进行解析。

路径以/flaskr开头的请求都将转发到127.0.0.1:8000，proxy_set_header可以添加请求头，让服务器知道发起请求的真实ip，在外部只能看到nginx服务器，看不到真实响应的服务器，实现了反向代理。

Location匹配规则

匹配前缀

  • 空: location后没有参数直接跟着URI，表示前缀匹配，代表跟请求中的URI从头开始匹配。
  • ~ : 执行一个正则匹配，区分大小写。
  • ~*: 执行一个正则匹配，不区分大小写。
  • ^~: 普通字符匹配，多用来匹配目录。
  • = : 执行普通字符精确匹配。
  • @ : “@” 定义一个命名的 location，@定义的locaiton名字一般用在内部定向，例如error_page, try_files命令中。它的功能类似于编程中的goto。

匹配顺序

  1. 寻找有没有“=”等号参数完全匹配的location，如果有完全匹配的等号location则停止匹配，执行该location中的指令，不去匹配其它类型的location。
  2. 匹配所有非正则表达式URI的location(包括空，=，^三种参数)。找到请求URI和location URI按前缀匹配最长的location，如果这个最长的location的参数是^，则停止匹配，执行该location中的指令，否则暂存该location。
  3. 匹配正则表达式URI的location(包括，*两种参数)，按location在配置文件中出现的顺序匹配，如果找到第一个匹配的locaiton则停止匹配，执行该location。
  4. 匹配完所有正则表达式都没有匹配的location，则执行第二步中暂存的最长前缀匹配location

Contextlib模块和生成器

上下文管理器

上下文管理器是指实现了__enter__方法和__exit__方法的对象，可用在with语句中。进入with代码块前自动调用__enter__方法，退出with代码块后自动调用__exit__方法，不论with代码块中是否发生了异常，__exit__方法都会执行。

__enter__方法返回一个对象，如果使用with... as f的语法会将返回值赋值到变量f上，在with代码块中可以使用这个变量。

__exit__方法接受三个参数，分别是exc_type，exc_value和exc_trackback，代表的是异常相关的信息，如果在with代码块中没有发生异常则都为None。__exit__函数具有屏蔽异常的功能，如果__exit__返回True，那么在with代码块中发生的异常会被屏蔽。

生成器

如果一个函数中有yield关键字，那么这个函数的返回值就是一个生成器。生成器相当于一个由用户控制的子程序，通过调用send方法后开始运行，每当运行到yield位置就挂起，生成器的运行状态继续保留，yield后面的表达式返回给调用者，再次调用send方法可以恢复上一次的运行状态，继续运行到yield语句处，如果生成器返回了一个值，则抛出StopIteration。

send方法

通过send方法运行生成器时，还可以发送一个值到生成器中，类似于函数的传参，在形如n=yield expression的语句中，n会被赋值为send方法发送的值，然后继续运行下一行。如果是第一次调用send方法，生成器还没被激活，send方法只能发送None,否则抛出TypeError。

通过send方法可以像函数一样操作生成器，比函数更优的是生成器能保存上一次运行的状态，当然闭包也能保存运行的状态，但是生成器更加灵活，语法更加简单清晰。

throw方法

throw方法可以向生成器中抛出异常，生成器可以捕获异常进行处理，也可以选择不处理异常，后续讨论的contexmanager修饰器就利用了这个方法将with代码块中异常传递到生成器中进行处理。

close方法

close方法用于关闭生成器，调用close方法后，再调用send方法只会抛出StopIteration，调用throw方法则抛出传入的异常。

generator是内置类型，close方法也是内建方法，底层使用c语言实现的，但是在Python标准库的collection.abc模块中，Generator抽象类型实现了close方法，可以参考

close方法本质上就是通过throw方法抛出GeneratorExit异常，生成器要么运行到return语句返回，抛出StopIteration异常，要么不捕获GeneratorExit异常直接抛出。如果生成器捕获了GeneratorExit并且没有抛出异常（例如遇到了yield语句），则在close中抛出RuntimeError，说起来不是很清晰，遇到这方面的问题可以结合上面的代码分析。

GeneratorExit

GeneratorExit是一个特殊的异常，提供了稳定可靠的信号关闭生成器（使用close函数才能真正可靠），一般不要去捕获GeneratorExit。通常我们捕获和处理的异常都是Exception及其子类，GeneratorExit并不属于Exception的子类，所以except Exception: ...不会捕获GeneratorExit异常，但是会被except: ...捕获。（更多的异常层次结构参考官方文档）

contextmanager装饰器

前面讲了with语句和生成器，在这里将二者联系起来了。contextmanager装饰器可以把一个生成器转换成上下文管理器（_GeneratorContextManager，后面简称GCM），也就是说把__enter__和__exit__的代码都写到一个生成器函数里，在通过contextmanager装饰，就转换成上下文管理器了，这种方式简化了创建上下文管理器的过程，减少了重复性代码。

生成器函数以yield为分界线，yield之前为__enter__函数的代码，yield之后为__exit__的代码，在GCM的中，__enter__和__exit__依次调用一次send方法就好了。前面还提到__exit__具有处理异常的功能，with代码块中的异常传递给GCR的__exit__方法，而在__exit__方法中，通过throw方法在传递给生成器函数，好了，现在处理异常的代码也能写在生成器函数中了。

TraceBack模块和Exception类

在python中，try…except用于捕获和处理异常，异常可能是由程序错误造成的，也可以通过raise语句手动抛出异常。traceback模块是一个格式化和打印异常及回溯的内建模块，一般不需要直接使用这个模块，本篇文章旨在探索traceback模块的一些细节。

sys 模块中的 exc_info 函数

当程序抛出异常时，可以通过try…except语句进行捕获，也可以通过调用sys.exc_info()函数来获取异常信息。sys.exc_info()返回一个三元数组，包括异常类，异常实例和traceback实例。

当通过raise语句抛出异常之后，或者在在except代码块中执行时，sys.exc_info()返回当前的异常，当处理完异常之后，sys.exc_info()返回None。在try…except as e结构中捕获异常之后，异常对象的属性中已经包含了这个三元数组的所有信息.

traceback 对象

traceback对象包含了出现异常的位置信息，主要是抛出异常的文件及行号(tb_lineno)，如果异常出现在函数中，会递归地记录函数内部抛出异常的位置。多个traceback对象通过tb_next属性以链表的结构储存。

在打印异常信息时，会递归地打印出所有的信息，通过这些信息，我们知道了出现异常的文件和行号，方便修复异常。当函数调用的层数较多时，打印的信息太多反而不容易排查异常，这时候我们可以通过自己捕获和处理异常，限制回溯的层数。

print_tb vs. print_exc

print_tb是一个与print_exc相似的函数，使用print_tb函数时需要传入具体的trackback对象，但是，print_tb(tb_obj)与print_exc()功能并不完全一样，两者是包含关系，print_exc打印的信息更多.

print_tb只打印对应异常的回溯信息，print_exc还会打印Exception.__context__中所有的回溯信息。

Exception.__context__属性是什么呢? 当捕获了异常(原异常)之后，在except代码块里，如果又产生了新异常，这时新异常的__context__属性就会被置为原异常。这么做是有原因的，因为原异常产生并被捕获了才会执行except代码块里的语句，所以新异常的产生也是由原异常导致的，如果原异常没有产生，根本不需要执行except代码块。因此需要将原异常记录下来以备以后溯源。下面是一个测试：

raise 关键字

raise关键字可以主动抛出异常，有时候我们还会见到类似与raise…from None的语句。将上面代码改成这种形式之后,发现print_exc函数不再打印__context__属性中的异常信息了。

事实上，异常对象中有四个特殊的属性：

  • __cause__：由raise…from e语句将__cause__置为e
  • __context__：在except代码块中，当处理异常时发生了新的异常，自动置为原异常
  • __supress_context__：布尔值，False表示打印与否取决于__contex__，True表示抑制__context__，打印与否取决于__cause__。默认为False，raise …from …语句可将其置为True
  • __traceback__：异常对应的traceback对象，包含发生异常的位置信息

单独的raise语句在except代码块中使用，表示直接抛出当前异常，如果在没有异常的上下文中，则抛出RuntimeError。

总结

一般不需要直接使用traceback模块，如果要记录程序运行中的错误信息，应当使用日志工具。但是在contextmanager对象中，__exit__函数使用了exc_type，exc_value，exc_traceback作为三个参数，需要在退出上下文时处理可能发生的异常，因此理解异常对象也有助于理解上下文管理器对象。
